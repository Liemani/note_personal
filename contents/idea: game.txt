# 식물을 키우는 것 같은 게임
식물을 키우거나 나만의 왕국을 건설하거나
약간 창세기 같은, "보시기 좋았더라" 같은 게임
이를 위해서는 단순히 글쓰기를 하거나, 그림 그리기를 넘어서
내가 만든 창작물이 서로 상호작용을 하여, 흐뭇한 경험을 선사해야 한다.
그렇다는 것은 변수만 존재하는 것이 아니라 변수간의 상호작용, 즉 method가 존재한다는 것이다.
이를 구체화하기 위해서는 크게 2 가지로 나뉘어질 것이다.
1. 유저는 data를 생성하면 프로그램에 정의된 method대로 동작한다.
2. 유저가 data도 만들고 method도 정의한다.
캐쥬얼한 유저는 1을 선호할 것이고, 헤비한 유저는 2를 선호할 것이다.
유저의 창작 활동을 장려하는 게임

cli에서 만들 수 있는 게임으로 이런 키우기류 시뮬레이션 게임을 만드는 것이다.
object들이 있고 모든 것들이 매 frame마다 동작하는 것은 아니다.
정해진 시간이 된 class의 instance들만 행동을 한다.
유저는 object와 상호작용을 할 수 있다.
상호작용을 통해 object의 구체적인 status를 확인할 수 있다.
object들은 서로 상호작용을 한다.
file system구조의 map을 오가면서 새로운 world를 오픈하는 방식은 어떨까?
와 씹,, 각 tile에 존재하는 object에 code를 입력해서 행동을 정의할 수 있는 것이지
새로운 map은 더 많은 code 입력이 가능하다는 뜻이고, 더 복잡한 작업을 처리할 수 있게 되는 거지
개다가 나는 하나의 object가 두 개의 ascii 문자를 가져야 정사각형으로 예쁠 것이라고 생각했는데,
오히려 그럼 ascii 보다는 한글을 쓰는 것은 어떨까?
각 타일은 한글 코드를 가지는 거지.
instruction을 만들어두고, 이를 사용하는 machine이 존재하는 것돋 재밌겠다.
옛날에 생각했던 hnh의 macro처럼
mindustry처럼 특정 조건을 만족할 때마다 더 고급 instruction을 주는 거지.
처음에는 instruction을 읽는 machine이 하나만 존재한다.
나중에는 machine을 여러 기 다룰 수 있게 된다.
더 많은 instruction을 처리하는 machine도 있으면 재밌겠다.
단, 자연 환경을 가공한다는 이미지에, 적을 막는 것이 아니라, 시간 제한 없이 어떤 조건을 만족하는 것이 목적이다.
따라서 그냥 시간만 주구장창 써도 되고, 아니면 목적 달성을 위해 더 효율적인 프로세스를 만드는데 투자해도 된다.
단 효율적인 프로세스가 궁극적인 목적이어서는 안된다.
유저의 창작 활동을 돕는데 집중해야 한다.
각 맵은 실제 directory 안에 파일이나 directory로 존재하도록 하자.
다른 유저의 데이터를 읽어 마치 다마고치 처럼 상호작용이 가능하도록 하자.
결과가 "힘을 합쳐 용을 물리쳤습니다!"라던가, 서로 win-win하는 결과를 만들어서 서로의 상호작용을 장려하자.
하나의 root 파일이 담고 있는 모든 내용을 종합하여 하나의 생물처럼 다루자.

그 다음은 복사 붙여넣기 기능을 추가하고
현실의 무언가를 모방하도록 이미지화 한다.

# 게임 세상과의 상호작용의 종류
1. 물체를 향한 후 상호작용 키를 누른다
2. 물체와의 거리가 일정 거리 안에 들어오면 상호작용
3. 물체와 시야의 각도가 일정 각도면 상호작용
4. 1틱마다 특정 상황을 판단하는 상호작용
5. 다양한 상호작용에 반응하는 연쇄작용
6. 특정 변수에 따른 상호작용
