MVVM 은 view model 과 model 을 분리함으로써
model 과 별개로 view 를 자유롭게 다룰 수 있는 건가?
그래도 view model 안에는 view 만 바꾸는 함수와 model 을 바꾸는 함수가 공존해야 할 것이다
필요에 따라 필요한 함수를 호출해야 할 것이다
그리고 logic 이 model 에 들어간다고 하는데
그렇다는건 여기서 말하는 모델이 데이터만이 아니라는 것이다
그렇기 위해서는 model 이 view model 을 위한 실제를 가져야 한다는 것이다
그러면 view model 과 model 에 구조를 위한 코드가 중복될 것 같다
나라면 model 에 logic 을 넣지 않을 것이다
model 은 element 로 한정하고 구조를 빼면
view model 만 구조를 갖게 될 것이다
그러면 logic 도 view model 에서 처리되야 한다
이로 인해 view model 에 생긴 변화는 적절하게 model 에 전달된다
단, logic 이 view model 에 존재하는 경우 view model 의 부피가 커지니
logic handler 를 따로 둬서 처리를 모듈화하고 위임한다
이것들을 굳이 logic handler 로 빼지 않고 함수 하나로 처리하도록 해도 괜찮을 것 같다
그걸 기본으로 하고 game object 마다 가지는 특별한 case 만 interact handler 같은걸 만들면 괜찮겠다

    청크 변경

2023-06-13
디자인 패턴을 잘 설명하기 위해서는 각 개념의 의도를 명확히 해야 한다
또한 예시를 들기 위해서는 코드보다는 글을 써서 설명하는 것이 좋겠다
가령 MVC 모델을 설명한다면
model 은 데이터들을 다루는 객체들의 추상 개념
view 는 display 에 출력될 객체들의 추상 개념
controller 는 view 를 다룰 때 model 도 아니고 view 도 아닌 것들을 기술한 집합
새로 앱을 열었을 때 기존에 저장된 데이터를 뷰에 출력한다면
컨트롤러가 모델한테 기존에 저장된 데이터를 요청하고
받은 데이터를 뷰한테 출력을 요청한다
결국 이런 설명은 interface 와 implementation 으로 나눠서 설명해야 한다
interface 로 의도를 설명하고
implementation 으로 실제 적용 패턴을 설명한다

# 선
선을 일반화된 공식으로 묘사하기 위해 이렇게 하면 좋을 것 같다
내가 지금 말하고 싶은 선은 어떤 도구로 굽지 않게 쭉 그은 선이다

선은 진행 방향이 있다(azimuth)
선은 속도 혹은 굵기가 있다(strength)
선은 걸린 시간이 있다(time)

어떤 방향으로 얼마의 세기로 얼마의 시간 동안 그리느냐에 따라 다른 선이 된다
단 현실적으로 보자면 이것은 찰나의 측정을 위한 단위일 뿐이다
현실의 선은 방향, 굵기, 시간이 모두 변수다
즉, '선 중 특정 시간에서의 점 = 선(방향, 굵기, 특정 시간)' 이다

# 성공을 상상하라?
어떤 사람은 성공하려거든 성공한 자신의 모습을 상상하라고 한다
진짜 지금 당신이 원하는 성공을 했다고 현실인 것처럼 느껴야 한다고 말이다
이 주장은 어느 정도 맞다고 생각한다
내가 원하는 진정한 성공이 무엇인지 마음 깊이 느껴본다면
현실이 성공이 아니라는 괴리감을 이해할 수 있으니
성공하지 못한 현실에 안주할 수 없게 될 것이다

하지만 가상의 성공이 너무나 현실인 것으로 여겨 안주하게 된다면 어떨까?
성공을 너무나도 상상한 나머지 이미 성공한 것으로 여겨 내가 무엇이 모자란지 모르는 것이다
이미 나는 충분한 경지에 올랐다고 느껴 현실에서 움직여야 할 이유를 찾지 못하게 될 것이다

이 둘을 가르는 기준은 얼마나 디테일하게 성공을 상상했느냐 하는 것일 것 같다
추상적으로 성공을 상상하는 것은 나의 부족함을 눈치채지 못하게 하고 성공의 가치를 하락시킨다
반면 구체적이고 꼼꼼하게 성공을 상상한다면
자신의 부족함을 낯낯히 알게 되고 성공이 얼마나 만만치 않은 녀석인지를 알게 될 것이다

# 어제 답글의 feedback
어제 답글에 imoji가 하나도 달리지 않았다!
내 답글의 무엇이 마음에 안들은 것일까?
반복해서 읽다보니 '구현'이라는 단어와 'header'라는 단어가 적절하지 않았을 수 있다는 생각이 들었다.
찾아보니 '구현'과 '정의'는 미묘한 차이가 있었다.
	- 구현: interface의 짝, 대게 어떤 interface를 정의한 뒤 implementation을 구현한다. 이 때 implementation을 구현하는 작업에는 어떤 method를 정의하는 과정이 포함될 수 있다.
	- 정의: 어떤 변수나 함수의 구체적 내용을 명시한 것
구현이라는 표현보다는 정의가 더 적절하다.
header라는 단어는 header file과 function header 두가지로 해석될 수 있다.
만약 '구현' 이라는 단어를 사용하지 않고 '정의'를 사용했다면 그나마 header가 function header로 해석됐을 것이다.
구현과 header가 만나 뜻이 더욱 불분명해지면서 내 의도를 전달하기 어려워지게 됐다.
어제 답글이 어느 정도 문제 제기를 하는 데는 성공했지만 머리에 닿는 구체적이고 명확한 의미 전달을 하는 데는 실패했다.
이 사람들이 왜 이모지 안달아주지? 이모지에 인색하네? 라는 생각도 했었지만 지금 이렇게 내 글의 문제를 파악할 수 있게 되어서 오히려 이모지를 달아주지 않은 것이 좋은 피드백이라고 생각한다.
글을 작성할 때 정확한 단어를 사용하는 것이 중요하다.
내가 어제 작성한 답글에서 '구현'이라는 단어는 내 머리 속에만 있는 단어였다.
그래서 내가 읽기에는 문제가 없었지만, 다른 사람들이 읽기에는 감점 요소가 있는 글이었다.

# 열쇠와 자물쇠와 명작과 분산 투자
어떤 object를 가치 판단 함수의 인자로 넣어 가치를 반환받을 때, 가치에 영향을 미치는 다양한 판단 요인이 존재한다.
사람마다 각 판단 요인에 다른 가치를 둘 것이다.
가령 사람A는 명작을 판단할 때 스토리 > 논리적 타당성 > 표현 > 스릴 > 액션 등으로 가치를 매길 수 있고,
사람B는 스릴 > 액션 > 표현 > 스토리 > 논리적 타당성 으로 가치를 매길 수 있다.
이 때, 가로 축을 가치 판단 요인으로 하고 세로 축을 가치로 설정하면 개개인의 가치 판단 그래프를 그릴 수 있다.
이것을 자물쇠라고 하고 각 작품의 가치 요인 - 가치 그래프를 열쇠라고 한다면,
'자물쇠에 맞는 열쇠 == 특정 사람이 최고로 치는 명작'이라고 할 수 있다.
그럼 다음 주제인 분산 투자로 넘어간다.
작품을 생산하는 제작자의 입장에서는 액션 하나에만 몰두한다면 액션의 가치를 높게 치는 하나의 자물쇠를 열 수 있는 열쇠를 만드는 것이다.
그러면 많은 자물쇠를 풀 수 없다.
따라서 많은 자물쇠를 풀기 위해서는 다양한 가치 판단 요인에 분산 투자한 열쇠를 만드는 것이 좋을 것이다.

# 내 성격
난 의외로 감정을 밖으로 잘 드러내는 성격인 것 같다.
기분이 좋을 때나 기분이 나쁠 때 그것으로 목소리 톤과 억약이 바로 바뀐다.
그런데 역설적이게도, 내 기분에 대한 주관적 생각을 밖으로 잘 드러내지 않는다.
따라서 내가 기분이 나쁠 때 상대방은 내가 왜 기분이 나쁜지 알 수 없다.
그러면 상대방도 기분이 나빠지게 되고, 결과적으로 전체적인 팀 에너지가 감소한다.
해결방법은 2가지이다.
1. 내가 왜 기분이 나쁜지 스스로 확실히 인지하여 밖에 알리는 것이다.
2. 당장 기분이 나쁠 수 있어도 분명 관점에 따라 썩 불쾌한 일이 아닐 수 있다. 따라서 항상 좋게 생각하도록 노력하며 에너지를 뿜도록 한다.
아직은 생각만 한 것이므로 실제 선택에 따라 어떤 영향이 있을지는 간접 경험을 하거나 직접 경험을 해보아야만 할 것이다.

# function overloading
- c++에서는 function overloading을 지원해서 같은 이름에 다른 type을 가지는 함수를 implement하여 사용할 수 있다.
	- 같은 이름에 같은 type을 가지는 함수도 overloading을 지원했으면 좋겠다.
	- 이 경우 입력값이 특정 상수인지를 통해 호출할 함수를 구분하면 된다.
	- 가령 입력값이 n이면 myFunction(input = n)을 호출하고 입력값이 m이면 myFunction(input = m)을 호출한다.

# 기억
내가 언제 밤에 어떤 단어를 아침에 기억해낸 것은 다양한 것들과 연관시켜서 기억하기 위해 노력해서 가능한 것이었다고 생각하는데, 그것은 혹시 물리적으로 긴 시간동안 의식적인 생각을 한 것이 근본적인 원인인 것은 아닐까?
다음 번에는 다양한 관점에서 기억하려 노력하는 것이 아니라 같은 관점이라도 물리적으로 긴 시간 동안 의식적으로 기억하려고 노력해 보고, 그 결과를 비교해보자.
근대 자연스럽게 생각해보면 근본적으로 오래 기억하기 위해서는 전자의 방법이 맞을 것 같다. 한 가지 관점으로 긴 시간 의식적으로 생각을 하는 것은 투자한 시간에 비해 기억에 잘 남을 것 같지 않다.
뇌도 같은 전기 자극만 반복될 뿐이지 않은가! 다양한 관점에서 생각을 하는 것은 side effect로 긴 시간 동안 생각하게 되는 것일 뿐, 다양한 관점에서 생각을 하는 것이 다양한 전기 자극이 활성화되서 동일 시간동안에 더 많은 것을 기억할 수 있을 것 같다.

최초의 새로운 생각에 다시 접근할 수 있다.

내 생활 패턴이 존재하는데

나는 평소 하지 않을 행동을 하게 된다.
평소 하지 않을 행동은 안정성이 확보되어 있지 않다.
따라서 risky하다.
누군가는 이 risk를 감수하고 싶지 않아서 기존의 생활 방식을 고수한다.
물런 low risk로 예측 가능한 결과를 보장하는 것이 꼭 필요한 경우가 많다.
하지만 여건이 될 때마다 risk를 감수하고 새로운 행동을 하는 것도 좋다.
이 때 가장 좋은 것은 risk를 먼저 예측하고 잘 대처하는 것이다.
하지만 경험이 없는 risk를 먼저 예측하고 대처하는 것은 불가능에 가깝다.
따라서 경험이 없는 경우에는 risk를 최소화하여 경험하도록 하는 것이 중요하다.
그러면 작은 실패로 어떤 risk를 경험할 수 있다.
나는 얼마 전에 본가에 내려갔을 때 물건의 배치가 달라진 적이 있었다.
내 방 배란다로 가는 통로에 이삿짐이 쌓여 나가기 힘들었다.
이 때 평소 하지 않을 자세로 양 옆의 책장과 옷장의 모서리를 잡고
짐을 뛰어 넘으려고 했다.
그러다가 오른쪽 어깨 뒤쪽의 인대가 살짝 늘어났는데,

# 평가
오늘 snpark님과 hyepark님의 평가가 됐다. 오랜 기간 평가를 안하다가 snpark님의 평가를 하게 돼서 조금 부족하게 평가를 진행한 것 같다. 그리고 서로의 성격이 특정 대화 구도에서는 잘 맞았지만 특정 상황에서는 서로 제대로 이해하지 못하고 상황이 진행되고 있는 것은 아닌가? 하는 생각이 들었다.
한 번 평가자를 하고 좀 감을 찾아서 hyepark님과는 평가가 재밌게 진행됐다. 말씀을 잔잔히 하시는 분이라 나도 말이 잔잔해졌다. 그리고 평가 내용이 우연히 동일해서 이전 평가에서는 제대로 이해하지 못하고 시작해서 내가 질문도 많았고 오래걸렸었는데, 두 번째 평가라서 하시는 말씀을 내가 쭉 들을 수 있었다. 그리고 내가 질문을 던질 때도 더 이해가 된 상태에서 질문을 던질 수 있어서 오가는 내용이 짧고 간결했다. 그렇게 주어진 시간동안 충분히 원하는 대화를 주고 받은 뒤 평가를 마무리 할 수 있었다.
마지막 질문으로는 "더 말하고 싶은 부분이 있으세요?" 정도가 적절한 것 같다. "더 궁금하신 부분이나 물어보고 싶은 부분 있으세요?"라는 질문은 별로 반응이 좋지 않았다. 아무래도 상대를 낮추는 대화의 느낌이 드는 것 같다. 그래도 중간 쯤에 상대의 대화를 이끌어내기 위해 "진행하면서 애매했던 부분이 있으셨나요?" 정도는 괜찮은 것 같다.
그리고 어떤 개념에 대한 의견을 주고 받을 때도 상대가 잘못을 한 것처럼 말하는 것은 좋지 않다. "이건 왜 이렇게 안했어요?"라는 질문 보다는 "이렇게 해도 되는데 그렇게 한 이유가 무엇일까요?"가 좋다. 이때 톤은 너무 조용하게 하여 무서운 분위기를 만들지 말자. 너무 톤을 높여서 상대를 압박하는 느낌이 들게 하는 것도 좋지 않다. 이렇게 단어와 문장을 잘 선택해서 말을 하는 것도 훌륭한 기술이지만, 단순한 내용 이외의 높낮이, 크기 등을 통해 은연중에 상대의 감상을 컨트롤하는 것도 중요한 기술인 것 같다.
나는 내가 잘 아는데 상대가 잘 모르는 부분을 알려주는 기술이 부족한 것 같다. 때대로 상대의 기분을 잘 고려하지 않고 내가 하고 싶은 말을 쭉 쏟아놓는다. 하지만 아무래도 내 말만 하기보다는 의문형 문장을 한 번씩 주고 받는 것이 호응이 좋았다. 하지만 내가 애매하게 알고 있는 부분에 대해 상대에게 질문을 할 때 생기는 문제일 수도 있다. 애매한 부분에 대한 질문을 하다보니 질문 자체가 잘 전달되지 않았고, 내가 의도한 질문을 제대로 전달하기 위해 이후 추가적인 대화를 이어나가야만 했다. 이는 분명히 아쉬운 부분이다. 하지만 그럼에도 불구하고 애매한 부분을 잘 질문하는 방법이 존재할 것 같다. 그럴 때는 상대가 내 질문을 잘못 이해하고 대답해줄 때 내가 애매하게 알고 있어서 제대로 된 질문을 던지지 못해 상대가 제대로 된 질문을 이해하지 못한 상황임을 말하는 것이 좋을 것 같다.
나는 열심히 공부해서 평가받는 사람들에게는 더 혹독하게 물어보려고 노력한다. 하지만 단순히 혹독하게 물어보는 것과, 상대방이 노력한 내용을 평범한 노력과 차별하여 포인트있게 물어보는 것은 다른다. 혹독하게 물어보는 것은 상대의 기분을 썩 좋지 않게 만든다. 하지만 노력한 내용에 대해 날카롭게 물어보면 상대는 매우 신나한다. 그리고 그렇게 혹독하게 물어본 후 좋은 평가를 주면 상대는 썩 기분 좋아하지 않는다. 자신의 결점을 열심히 파고든 주제에 평가를 좋게 주면 인지부조화(?)가 일어난다고 볼 수 있을 것 같다. 때문에 첫째로, 상대의 노력을 날카롭게 물어보는 좋은 질문을 하도록 노력하고, 그 이상을 요구하는 질문을 하여 상대가 잘 대답하지 못했다고 한다면, 좋은 평가를 줄 때 "이런이런 점이 좋아 좋은 점수를 드립니다"라는 이유를 설명해줘야 한다.
# 자기효능감/현실
자기효능감은 현실을 구체적으로 인지하지 않고 어느 정도 추상화된 상태로 이해하여 "내가 할 수 있을 것이다"라고 느낄 때 높아진다고 생각한다.
만약 진실을 알게 되어 1만시간의 노력을 해야됨을 이해하고 수만 페이지의 글을 읽어야 함을 이해하는 등, 현실을 이해하면 자기효능감이 떨어질 수도 있을 것 같다.
나는 현실을 알려주는 방식을 선호한다.
그럴 경우 일반적인 사람들은 대화 후 굉장히 기죽고 일을 하기 싫어하는 상태가 된다.
이건 꽤 악효과인 것 같다.

하지만 너무 현실을 고려하지 않고 자신감만 가득 찬 경우도 있다.
이 경우는 '건전하게' 현실을 직시할 수 있도록 도와주는 것이 좋다.

즉, 이 경우도 밸런스의 문제인 것 같다.
한 쪽에 치우치지 않도록 해주는 것이다.
중간을 유지하도록 도와주는 것이 아니라 한 쪽에 치우치지 않도록 도와주는 것이 중요한 것 같다.
중간을 유지하도록 도와주는 것은 manage이고, 한 쪽에 치우치지 않도록 도와주는 것은 lead이다.
사람은 사람을 manage하지 않아야 한다.
가능하지도 않고 비효율적이다.
사람은 사람을 lead해야 한다.
한 쪽에 치우치지 않도록 방향을 제시해주고 구체적인 것은 상대를 믿고 맡기는 것이다.
따라서 "현실은 이럴거 같에"라며 구체적인 생각은 상대가 하도록 유도하는 것이 좋다.
"현실은 이렇고 저렇고 어떻고"하는 것은 지양하자.
하지만 현실은 이렇고 저렇고 어떻고를 논하는 것은 상당히 재미있는 일이다.
이 재미있는 일은 해당 사안에 대한 지식 수준이 맞는 사람과 하자.

- 우선 드리고 싶은 말씀
"하기 싫어지는 것"는 언제나 발생할 수 있는 일상적인 문제죠.
저마다 이 문제가 발생했을 때 해결하기 위해 취하는 행동이 다를 것 같은데요.

제가 우선 드리고 싶은 말씀은,
"이 문제가 발생할 경우 다시 하고 싶어질 때까지 일주일이고 논다"라는 선택은 하지 않으셨으면 좋겠습니다.
이 선택은 효율이 좋지 못하다고 생각합니다.
게다가 이 선택은 실제 문제에 대응하여 환경에 변화를 주는 것이 아닙니다.
그리고 환경에 변화를 주지 못하면 다음에 같은 문제가 발생했을 때 또 같은 선택을 하게 될 확률이 높다고 생각합니다.
저는 이 일상적인 문제를 맞닥뜨릴 때마다 새로운 시도를 해보는 것을 추천드립니다.

제 경험을 말씀드리면,,
저는 집에서 혼자 독학을 시도하던 때가 있었는데,
사실 반 이상은 게임하고 유튜브 보기 일쑤였습니다.
그러면서 "같은 환경에서 자기 통제를 통해 더 나은 행동을 하는 것은 너무 힘들고 비효율적인 행위이다."라는 것을 깨닳았습니다.
아마 가끔 어쩌다 다른 환경에 발을 들일 때
평소에는 보기 드문 '열심히하는 나'를 발견하면서 이런 사실을 깨닳은 것 같습니다.
"환경이 자기 통제보다 중요하다"
저는 이때부터 환경의 중요성을 인지했습니다.(통제가 중요하지 않다는 말은 아닙니다!)

- 나의 3 가지 방법
1. 환경 변화주기
과거 저는 우연히 집 밖으로 나가 '다른 사람들이 있는 곳'에서 공부를 하는 경험을 하게 됐습니다.
그리고 저는 '다른 사람들이 있는 곳'에 있으면 열심히 공부한다는 사실을 인지했습니다.
그래서 지금은 가급적이면 그런 곳에 가 있으려고 노력합니다.
물론 그런 와중에서도 집에서 혼자 있을 때도 열심히 공부하도록 만드는 방법을 계속 찾습니다.
저에게 효과가 있는 방법 중 하나는 42 디스코드 채널의 접속중인 사람 목록을 띄워두는 것이었습니다.

혼자 공부하는 것을 좋아해서 혼자 있으면 '다른 사람이 있는 곳' 버프를 받지 못해 효과거 적었는데,
디스코드 채널의 접속중인 사람 목록을 띄워두면 100명 이상이 접속해 있다고 보이고
누구는 VSCode 작업중이라고 뜨고 아는 사람도 보이고 해서 '다른 사람이 있는 곳' 버프를 좀 받았습니다.

2. 롤모델 찾아보기
예를 들어, 전에 어떤 분이 슬랙에도 올려주셨던 영상인데

마지막 Q&A에서도 말씀을 잘 하시더라구요.
"하기 싫어지는 문제"가 발생할 때 한 번씩 보면 저는 효과가 있더라구요.

3. 하지 않아도 될 일을 기록하고 의식적으로 하지 말아야 한다고 생각하기
이 방법은 직접적으로 하기 싫은 것을 하도록 도와주지는 않지만
하지 않아도 될 일을 '시작'하지 않는데 도움을 줍니다.
저는 개인적으로,
'지금 할 필요가 없는 일'인 게임과
'지금 해도 미래의 나에게 크게 도움이 되지 않는 일'인 유튜브 시청에
많은 시간을 소모하는 사람입니다.
밥먹을 때 유튜브를 보기 시작하고 계속 보다보면 자기 전까지 유튜브를 보는 경우도 있었습니다.
그래서 이 방법을 통해 이런 일을 시작하는 것을 의식적으로 막는 것이 효과가 좋았습니다.
추가적으로, 이 방법을 통해 집에서도 놀지 않고 공부하는 시간이 비약적으로 늘었습니다.

- 자신만의 해결 방법
이렇게 제가 경험해온 저만의 해결 방법들을 적어봤는데요.
저의 해결 방법은 제가 저를 위해 시도하고 정리한 방법이기 때문에
다른 분들에게는 맞을 수도 있고 맞지 않을 수도 있습니다.
따라서 자신에게 효과가 큰 자신만의 해결 방법을 찾으셨으면 좋겠습니다.

그리고 문제가 발생하지 않으면 해결 방법을 시도해볼 수 없습니다.
'뭔가 하기 싫어질 때'는 오히려 그런 상황을 극복할 다양한 방법을 시도해 볼 좋은 기회라고 생각해볼 수도 있습니다.
이 기회마다 기존에 생각해 둔 해결 방법을 하나씩 적용해 보면서
나만의 해결 방법을 찾아 나간다면
분명 오늘의 나는 어제의 나보다 더 발전해 있을 것입니다.

- 끝으로
저는 문제에 대응하지 않고 피하는 선택을 하면서 긴 기간을 보냈습니다.
직접 겪지 않은 역사를 공부하여 더 좋은 미래를 만들기 위해 사용하듯이
저의 경험이 더 좋은 미래를 만드는데 도움이 됐으면 좋겠습니다.

# 무엇이 '나' 인가?
동일한 한 사람이기 위해서는 function, input, output 이 동일해야 한다. 사람으로서 function 은 뇌와 온몸의 기억 세포 등이 될 것이고 input 은 오감 등의 감각 기관을 통해 받아들이는 정보, output 은 신경계로 전파되는 전기 신호이다.
'만약 뇌를 기계로 옮기면 같은 사람으로 봐야 하는가?' 라는 질문을 한다면, function 은 동일하지만, copy 한 시점으로부터 input 과 output 이 다르기 때문에 서로 다른 지성체라고 할 수 있다. 게다가 function 의 구조는 같지만 구현이 다르다는 차이점도 있다.(물리적으로 다른 물질로 구성됨)
그리고 function, input, output 을 통해 '나' 를 정의했다면, 내 몸의 일부가 사라지는 것도 function, input, output 에 변화가 생긴 것이기 때문에, 사라지기 전의 나와 사라진 후의 나는 다른 지성체라고 할 수 있다. 다른 지성체이기 때문에 같은 외부 환경에 다른 선택을 하는 상황이 존재할 수 있다.
만약 뇌의 일부가 사라진다고 해도 같은 맥락으로 이해할 수 있다.

# [Common Instructions]
malloc 이 NULL 을 반환하는 경우에 segfault 등이 발생하지 않도록 하는 것은 해야 한다지만, dup(), pthread_mutex_init() 등의 함수가 에러로 인해 오작동하는 경우도 굳이 고려해야 할까요?
개개인이 이에 대한 답이 다를 수 있을 것 같은데요. 다양한 의견 답글로 달아주시면 감사하겠습니다!

대부분 과제에 있는 Common Instructions 를 보면 segfault, bus error 등으로 인해 **예상치 못하게** 프로그램이 종료되지 않아야 한다는 문장이 있습니다.
때문에 대부분 카뎃분들이 malloc() 은 반환값이 NULL 일 경우를 체크하여 비정상적인 작동 및 종료를 방지하는데요.
사실 에러로 인해 정해진 값을 반환하는 함수는 malloc() 뿐만이 아니라 라이브러리 함수와 시스템 콜 함수를 통틀어 굉장히 많습니다.
하지만 이런 함수에 대해 에러 가능성이 있다는 것을 어렴풋이는 알더라도 굳이 더 깊이 알아보고 적용하는 분은 보지 못했고, 에러 가능성이 있는 함수가 에러인 상황을 생각도 해보지 않은 경우도 많았습니다.
그 함수가 에러를 발생하여 특정값을 반환할 수 있다고는 하지만, 일반적인 환경에서는 에러가 발생하지 않기 때문에 굳이 저런걸 다 염두하고 코드를 작성해야 하는가 하고 생각하시는 분도 많은 것 같습니다.

저는 이 쓰레드로 인해 ‘에러로 인해 특정 값을 반환하는 모든 함수에 대해 굳이 더 생각을 해봐야하는가?’ 라는 지금까지는 불확실하고 애매했던 물음에 답이 되었으면 좋겠습니다.

# 함수의 분류와 추상화
함수는 어떤 입력에 따라 항상 어떤 출력을 한다
입력값은 다양하다. 시간, 함수의 상태를 나타내는 변수 등이 전부 입력 값이 될 수 있다
가령 플립플롭의 경우 트랜지스터의 개폐 상태가 해당 추상 함수의 인자로 들어가는 함수로 표현할 수 있다 O = f(s, r, t1, t2)
다만 특정 (t1, t2) 값은 절대 입력될 수 없다는 특징이 있다

automata 는 상태 변수와, 입력 값과 상태 변수로 새로운 상태 변수 값을 출력하는 함수를 둠으로써 내부 상태 변화를 묘사한다
M = (Σ, Γ, Q, δ, λ)
만약 Q 당장 들어온 값이 같아도 이전에 Q 값들에 영향을 받는다면 그 모든 Q 들을 하나의 Q 로 잡는 것이 맞을 것이다

automata 는 모든 함수를 위한 모델이 아니고, 기초적인 트랜지스터 등을 설명하기 위해 최적화 된 모델이다

# 사람의 변화
사람이 행동을 바꾸기 위해서는 2 가지 과정을 거쳐야 하는 것 같다

1. 자신의 특정 행동에 문제가 있음을 인지한다
2. 인지를 잊기 전에 문제가 재발한다
3. 우연히 또는 노력으로, 문제 없이 원활히 넘어갈 수 있는 대체 행동을 성공적으로 하였다 (3 번은 아직 검증이 덜 되었다. 1 , 2 도 검증된 것은 아니지만)

# 추상화와 구체화
추상화 -> 구체화 과정을 통해 구체적인 것을 구현할 수 있다
추상화는 스케치하는 과정이고, 구체화는 추상화된 것으로부터 실물을 만드는 과정이다
추상화와 구체화를 보조하는 프로그램이 있으면 좋겠다

# 추상화
추상화하는 작업은 프로그래밍에서만 필요한 것이 아니다
가령 글쓰기에서 특정 주제로 글을 쓰고자 할 때, 그 주제를 잘 추상화하여야 할 것이다
추상화한다는 것은 그 주제에 대한 영역을 명확히하고, 그 외의 것들과 구분이 되는 기준을 만들고, 주제의 역할을 구체화하는 것이다
추상화라는 것이 추상적으로 만드는 것인데, 그럼에도 불구하고 동시에 구체화를 하는 것이다
즉, 추상적인 개념을 구체화하여 실제 감각으로 느낄 수는 없지만 머리로 이해할 수 있도록 만드는 작업이 추상화 작업이다

사람간의 대화가 잘 되지 않을 때, 종종 특정 단어를 서로 다르게 추상적으로 인지하고 있는 경우가 있다
이처럼 추상화라는 것은 정답이 없기 때문에 같은 단어를 쓰더라도 사람마다 그 단어가 어떻게 추상화되어 있는가가 다 다르기 때문에 대화가 제대로 이루어지지 않는 경우가 존재하는 것이다

가령 삼각형을 추상화할 때
“세 개의 angle 을 가지는 도형’ 이라 할 수도 있겠으나
‘세 변으로 둘러쌓인 도형’ 이 널리 통용되는 삼각형의 정의이다
